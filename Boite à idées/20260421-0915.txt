# Multivers MMO – OTServ + Ryzom Core

## 1. Plan d’intégration IA → OTServ (The Forgotten Server)

### 1.1. Objectif

- **But:** valider ton orchestrateur IA (30k+ NPC, routines, gauges, événements).
- **Principe:** OTServ reste le “simulateur de monde”, ton orchestrateur IA pilote les NPC via une API maison.

---

### 1.2. Architecture minimale

- **OTServ (C++ + Lua)**
  - Gère : cartes, joueurs, NPC, combats, inventaires, persistance.
- **Service IA externe (Python/Go/TS, peu importe)**
  - Gère : décisions, plannings, comportements, états internes (gauge engine).
- **Bridge API**
  - Expose des endpoints type :
    - `POST /npc/{id}/intent`
    - `GET /world/snapshot`
    - `POST /events`

---

### 1.3. Étapes concrètes

#### 1.3.1. Récupérer et lancer OTServ

- **Repo conseillé :** The Forgotten Server (TFS).
- Étapes génériques (Linux) :

```bash
git clone https://github.com/otland/forgottenserver.git
cd forgottenserver
mkdir build && cd build
cmake ..
make -j$(nproc)
```

- Configurer `config.lua` (DB, ports, etc.).
- Lancer le serveur et vérifier qu’un client peut se connecter.

---

#### 1.3.2. Ajouter un module “bridge” côté serveur

- **But:** permettre à OTServ de parler à ton orchestrateur IA.
- Options :
  - **HTTP client léger** (libcurl) dans le C++.
  - Ou **socket TCP/UDP** vers un daemon IA.
- Exemple de flux :
  - OTServ tick → collecte les NPC “contrôlés par IA externe”.
  - Pour chaque NPC :
    - Envoie un `state` à l’IA (position, HP, contexte).
    - Reçoit une `action` (move, talk, attack, emote).
    - Applique l’action via les fonctions natives OTServ.

---

#### 1.3.3. Côté orchestrateur IA

- **Responsabilités :**
  - Maintenir un état interne par NPC (gauge engine, objectifs, mémoire).
  - Décider des actions à chaque tick ou sur événement.
  - Gérer les “rôles” (citoyen, marchand, garde, bandit, etc.).
- **API à exposer (exemple) :**

```text
POST /decide
Body: { npc_id, state_snapshot }
→ Response: { action: "move|talk|attack|idle", params: {...} }
```

---

#### 1.3.4. Scalabilité IA

- **Stratégies :**
  - Tick différencié (NPC “loin” = tick moins fréquent).
  - Batching : envoyer des groupes de NPC à la fois.
  - Prioriser les NPC proches des joueurs.

### 3.1. Vision

    - Gestion des NPC (identité, background, objectifs).
    - Gauge engine (Lyra).

- **Connecteur OTServ**

- **Connecteur Ryzom Core**

👉 **Résultat :** tu peux développer ton IA une fois, et changer de moteur sans tout réécrire.

---

### 4.2. Architecture serveur

- **OTServ**
  - Monolithique mais lisible.
  - Boucle de jeu simple.
  - NPC scriptables en Lua.
  - Facile à hooker pour ajouter un bridge IA.

- **Ryzom Core**
  - Architecture en services :
    - Login, shard, AI, world, etc.
  - Basé sur NeL (Nevrax Library).
  - Beaucoup de couches (réseau, entités, gameplay, outils).
  - Plus proche d’un vrai MMO commercial.

---

### 4.3. Performance & scalabilité

- **OTServ**
  - Très léger.
  - Peut gérer beaucoup de NPC (surtout si IA externe).
  - Limité par la représentation 2D et les mécaniques simples.

- **Ryzom Core**
  - Pensé pour un MMO 3D.
  - Réseau optimisé, gros mondes.
  - Plus lourd à scaler, mais plus réaliste pour un jeu final.

---

### 4.4. Extensibilité & modding

- **OTServ**
  - Scripts Lua très simples.
  - Idéal pour expérimenter des systèmes de jobs, routines, quêtes.
  - Limité pour des features avancées (véhicules, housing 3D, etc.).

- **Ryzom Core**
  - Très extensible, mais demande de comprendre l’architecture.
  - Permet :
    - nouveaux systèmes de skills,
    - nouveaux types d’entités,
    - nouveaux systèmes de gameplay (professions, housing, etc.).

---

### 4.5. Alignement avec ton multivers

- **OTServ**
  - Parfait pour :
    - tester ton orchestrateur IA,
    - valider les comportements émergents,
    - simuler des sociétés massives.

- **Ryzom Core**
  - Aligné avec :
    - sandbox 3D,
    - professions,
    - crafting profond,
    - villes vivantes,
    - multivers à long terme.

# 🧩 2. Peut‑on tout mettre dans une seule VM ?

## ✔ Techniquement : **oui**
Tu peux mettre :

- OTServ  
- Ryzom Core  
- Orchestrateur IA  
- Base de données  
- Outils  

…dans une seule VM de **8 vCPU / 16 Go RAM**.

Ça va tourner.

## ✘ Stratégiquement : **non**
Voici pourquoi :

### ❌ 1. Les moteurs n’ont pas les mêmes cycles de vie
- OTServ = sandbox IA → tu vas le redémarrer souvent.  
- Ryzom Core = shard MMO → tu veux qu’il reste stable.  

### ❌ 2. Les charges CPU ne sont pas synchrones
- OTServ = léger, mais ton IA peut saturer le CPU.  
- Ryzom Core = sensible aux spikes CPU → lag si l’IA charge trop.

### ❌ 3. Tu veux isoler les crashs
- Si ton orchestrateur IA plante → Ryzom Core doit continuer.  
- Si Ryzom Core plante → ton IA doit continuer.

### ❌ 4. Tu veux scaler indépendamment
- OTServ peut rester petit.  
- Ryzom Core peut grandir.  
- L’IA peut devenir énorme.
